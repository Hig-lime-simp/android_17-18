# Weather Dashboard

## Описание приложения

**Weather Dashboard** — это Android-приложение, демонстрирующее работу с корутинами в Kotlin. Приложение имитирует загрузку данных о погоде из трёх источников (температура, влажность, скорость ветра) с использованием асинхронных операций, параллельной загрузки и обработки ошибок без блокировки пользовательского интерфейса.

## Функциональность приложения

- 🌡️ **Загрузка температуры** — имитация сетевого запроса (~2 секунды)
- 💧 **Загрузка влажности** — имитация сетевого запроса (~1.5 секунды)
- 🌪️ **Загрузка скорости ветра** — имитация сетевого запроса (~1 секунда)
- 🔄 **Параллельная загрузка** — все данные загружаются одновременно (~2 секунды вместо 4.5)
- ⚠️ **Обработка ошибок** — корректное отображение ошибок без краша приложения
- 📊 **Вычисление индекса погоды** — демонстрация тяжёлых вычислений на Dispatchers.Default
- ⏱️ **Автообновление** — автоматическая перезагрузка данных каждые 10 секунд

## Контрольные вопросы

### A) В чём разница между `launch` и `async`?

| Метод | Возвращает результат? | Когда использовать |
|-------|---------------------|-----------------|
| `launch` | Нет (возвращает `Job`) | Когда нужно просто выполнить задачу и не ждать результат (например, отправка аналитики) |
| `async` | Да (возвращает `Deferred<T>`) | Когда нужен результат выполнения для дальнейшей обработки |

**Пример кода:**
```kotlin
// launch — "запусти и забудь"
viewModelScope.launch {
    saveToDatabase(data) // просто сохраняем, результат не нужен
}

// async — "запусти и получи результат"
val result = viewModelScope.async {
    fetchDataFromApi() // возвращает Deferred<Data>
}.await() // ждём результат
```

### B) Что такое `suspend` функция?

**`suspend` функция** — это функция, которая может быть приостановлена и возобновлена позже, не блокируя поток выполнения.

**Может ли она вызываться из обычной функции?**
- ❌ Нет, `suspend` функция может вызываться только из другой `suspend` функции или из корутины (`launch`, `async`)

**Почему `delay()` не блокирует поток?**
- `delay()` — это `suspend` функция, которая приостанавливает корутину, а не поток
- Поток освобождается и может выполнять другие задачи
- После истечения времени корутина возобновляется на том же или другом потоке

```kotlin
// ❌ Блокирует поток (НЕ использовать в UI!)
fun blockingFunction() {
    Thread.sleep(2000) // поток "заморожен" на 2 секунды
}

// ✅ Не блокирует поток
suspend fun nonBlockingFunction() {
    delay(2000) // корутина приостановлена, поток свободен
}
```

### C) Зачем нужны разные диспетчеры?

**Диспетчеры (Dispatchers)** определяют, на каком потоке будет выполняться корутина.

| Диспетчер | Когда использовать | Пример задачи |
|-----------|-------------------|--------------|
| `Dispatchers.Main` | Обновление UI, работа с View | Изменение текста в TextView, показ диалога |
| `Dispatchers.IO` | Сетевые запросы, работа с файлами, БД | Загрузка данных из API, чтение/запись файлов |
| `Dispatchers.Default` | Тяжёлые вычисления, обработка данных | Сортировка больших массивов, парсинг JSON |

**Что будет, если выполнить тяжёлое вычисление на `Dispatchers.Main`?**
- UI "зависнет" на время выполнения вычисления
- Пользователь увидит фризы, приложение может получить ANR (Application Not Responding)
- Система может принудительно закрыть приложение

```kotlin
// ❌ Плохо: вычисления на Main потоке
viewModelScope.launch {
    val result = heavyCalculation() // UI зависнет!
    textView.text = result.toString()
}

// ✅ Хорошо: вычисления на Default потоке
viewModelScope.launch {
    val result = withContext(Dispatchers.Default) {
        heavyCalculation() // выполняется в фоне
    }
    textView.text = result.toString() // возвращаемся на Main для обновления UI
}
```

### D) Что произойдёт, если не обработать исключение в корутине?

**Без обработки исключений:**
- Исключение "всплывет" до ближайшего обработчика
- Если обработчика нет — приложение крашнется с `FATAL EXCEPTION`
- В Logcat появится сообщение об ошибке и стек вызовов

**Как корректно обрабатывать ошибки:**
1. Использовать `try-catch` внутри `launch/async`
2. Использовать `CoroutineExceptionHandler` для глобальной обработки
3. Использовать `supervisorScope` для изоляции ошибок дочерних корутин

```kotlin
// Пример обработки ошибок
viewModelScope.launch {
    try {
        val data = repository.fetchData()
        _uiState.value = UiState.Success(data)
    } catch (e: IOException) {
        _uiState.value = UiState.Error("Нет соединения с интернетом")
    } catch (e: Exception) {
        _uiState.value = UiState.Error("Произошла ошибка: ${e.message}")
    }
}
```

### E) Как работает автоматическая отмена корутин?

**`viewModelScope`** — это область жизни корутин, привязанная к `ViewModel`.

**Когда корутины отменяются автоматически:**
- При вызове `ViewModel.onCleared()` (когда ViewModel уничтожается)
- При закрытии экрана/активности
- При повороте экрана (ViewModel сохраняется, корутины продолжаются)

**Преимущества:**
- Не нужно вручную отменять корутины
- Нет утечек памяти
- Ресурсы освобождаются автоматически

```kotlin
class MyViewModel : ViewModel() {
    init {
        // Эта корутина автоматически отменится при уничтожении ViewModel
        viewModelScope.launch {
            while (true) {
                delay(1000)
                // какая-то периодическая задача
            }
        }
    }
    
    override fun onCleared() {
        super.onCleared()
        // Все корутины в viewModelScope уже отменены!
    }
}
```
